# OFERTA_PILOTO_Anclora_SyncXML_v0.1-baseline

# Piloto Controlado Anclora SyncXML — Revisión de huéspedes desde Excel

**Subtítulo técnico:**  
**De Excel/XLSX a datos revisados y XML de prueba revisable, sin prometer cumplimiento legal ni integración automática con SES.**

## Conclusión

La oferta piloto de Anclora SyncXML debe presentarse como una **validación controlada del flujo Excel/XLSX → revisión → detección de errores → XML revisable**, no como un producto final de cumplimiento, ni como integración oficial con SES.HOSPEDAJES, ni como sustituto de un PMS o una gestoría.

El objetivo del piloto es comprobar si Anclora SyncXML aporta valor real a pequeños alojamientos, viviendas turísticas o gestores que trabajan con Excel/XLSX, usando **datos sintéticos, anonimizados o muestras controladas**. No debe trabajarse con datos reales de huéspedes salvo que antes estén cerrados los aspectos de seguridad, RGPD, DPA, retención, validación técnica y responsabilidades aplicables.

## 2. Cliente ideal

El piloto está pensado para:

- pequeños alojamientos;
- viviendas turísticas;
- propietarios individuales;
- gestores con pocos inmuebles;
- alojamientos que trabajan actualmente con Excel/XLSX;
- equipos que gestionan datos de reservas y huéspedes de forma manual o semimanual;
- usuarios que no necesitan adoptar un PMS completo;
- clientes que aceptan trabajar en modo piloto controlado;
- clientes que pueden usar datos sintéticos, anonimizados o muestras reducidas;
- responsables que entienden que el piloto no equivale a cumplimiento legal garantizado.

### No es ideal para

El piloto no está pensado para:

- cadenas hoteleras;
- operadores multipropiedad complejos;
- negocios que exigen integración automática con SES desde el primer día;
- clientes que quieren delegar asesoramiento legal;
- clientes que solo aceptarían trabajar con datos reales desde el inicio;
- empresas que necesitan PMS integral, channel manager, CRM, facturación o automatización completa.

## 3. Problema que resuelve

### Dolor principal

> “Trabajo con Excel/XLSX, tengo datos de huéspedes que revisar y necesito una forma más clara de detectar errores antes de preparar un XML revisable.”

### Problemas específicos

El piloto aborda problemas operativos como:

- datos incompletos o inconsistentes;
- revisión manual lenta;
- duplicados difíciles de detectar;
- riesgo de copiar errores desde Excel;
- dificultad para revisar datos antes de generar XML;
- dificultad para entender un XML generado;
- exposición innecesaria de datos personales sensibles;
- necesidad de ordenar el flujo de huéspedes sin implantar un PMS completo.

### Matiz importante

El piloto **no resuelve por sí solo el cumplimiento legal**, no garantiza aceptación por SES.HOSPEDAJES, no sustituye el criterio del responsable del alojamiento y no reemplaza asesoramiento legal, PMS o gestoría.

## 5. Qué no incluye

El piloto no incluye:

- garantía de cumplimiento legal;
- asesoramiento jurídico;
- promesa de evitar sanciones;
- integración oficial con SES.HOSPEDAJES;
- envío automático al Ministerio del Interior;
- uso en producción con datos reales;
- sustitución de PMS, gestoría o asesoría;
- almacenamiento permanente de datos personales;
- pre-check-in productivo;
- multipropiedad avanzada;
- API;
- roles avanzados;
- channel manager;
- importadores directos de Booking, Airbnb, Lodgify u otros canales;
- validación XSD estándar completa si todavía no está implementada en el entorno del piloto;
- claim de XML aceptado por SES si no existe respuesta documentada de preproducción.

Esta exclusión está alineada con la regla de prudencia del proyecto: no afirmar integración real con SES si no está demostrada y no recomendar uso con datos reales si existen riesgos pendientes de seguridad, privacidad, RGPD, validación o persistencia.

## 6. Condiciones de seguridad y privacidad

El piloto debe ejecutarse bajo estas reglas:

1. Sin datos reales por defecto.
2. Uso preferente de datos sintéticos, anonimizados o muestras controladas.
3. No cargar documentos reales de huéspedes, pasaportes, DNI, emails, teléfonos o direcciones completas.
4. No activar persistencia permanente salvo que existan garantías cerradas.
5. No almacenar Excel completo ni XML completo con PII real.
6. Mantener preview enmascarada.
7. Mantener revisión humana obligatoria.
8. Registrar solo eventos técnicos sin PII cuando aplique.
9. Borrar los datos de operación al finalizar la prueba.
10. No usar resultados del piloto para comunicaciones oficiales.

El piloto debe tratarse como una prueba controlada de flujo y utilidad, no como operación productiva.

## 7. Uso de datos sintéticos, anonimizados o muestras controladas

### Orden recomendado

| Tipo de dato | Uso recomendado |
|---|---|
| Datos sintéticos | Opción preferente para demo y validación inicial. |
| Datos anonimizados | Útiles para probar la estructura real del Excel sin exponer PII. |
| Muestras controladas | Solo si han sido revisadas para eliminar o sustituir datos personales. |
| Datos reales | No recomendados hasta cerrar RGPD, DPA, retención, seguridad, validación técnica y responsabilidades. |

### Ejemplo de anonimización esperada

- Sustituir nombres reales por “Huésped 1”, “Huésped 2”.
- Sustituir documentos por valores ficticios válidos.
- Sustituir emails por direcciones de prueba.
- Sustituir teléfonos por números ficticios.
- Sustituir direcciones por datos genéricos.
- Mantener estructura de columnas y formatos para probar el flujo.
- Eliminar observaciones, notas internas o campos libres que puedan contener PII.

## 11. Criterios de éxito

### Criterios técnicos y operativos

El piloto se considera exitoso si:

1. El cliente puede entregar una muestra sintética o anonimizada representativa.
2. Anclora SyncXML puede importar el XLSX o identificar claramente qué impide importarlo.
3. Se detectan errores, duplicados o campos incompletos de forma comprensible.
4. El cliente entiende qué debe corregir antes de generar XML.
5. Se genera un XML revisable de prueba.
6. El cliente considera útil la revisión antes del XML.
7. Se documentan límites, bloqueos y siguientes pasos.
8. No se usan datos reales sin condiciones cerradas.
9. No se formula ningún claim de cumplimiento legal ni aceptación SES no demostrado.

### Criterios de aprendizaje comercial

El piloto también debe servir para validar si:

- el cliente entiende la propuesta en menos de 5 minutos;
- el cliente reconoce el problema como real;
- el cliente estaría dispuesto a pagar por revisión, preparación y reducción de errores;
- el cliente acepta que Anclora SyncXML no sustituye asesoría legal ni PMS;
- el cliente identifica qué parte del flujo le aporta más valor: revisión, errores, XML, privacidad o futura asistencia.

## 12. Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| El cliente envía datos reales sin avisar | Protocolo claro: solicitar datos sintéticos o anonimizados y rechazar PII real si no hay condiciones cerradas. |
| El cliente interpreta el XML como aceptado oficialmente | Disclaimer visible: XML revisable, no aceptación oficial. |
| El cliente espera envío automático a SES | Aclarar exclusión desde la propuesta. |
| El Excel del cliente no encaja con el parser actual | Tratarlo como aprendizaje y documentar necesidad de mapeador visual. |
| El cliente espera asesoramiento legal | Derivar a asesoría legal externa. |
| La demo funciona solo con un formato concreto | Presentar el piloto como validación de encaje, no como producto universal. |
| Se confunde privacidad por defecto con cumplimiento RGPD completo | Aclarar que privacidad técnica no sustituye DPA, textos legales, retención ni base jurídica. |
| El cliente quiere pasar a producción demasiado pronto | Condicionar cualquier paso posterior a seguridad, RGPD, DPA, retención, XSD, SES, QA y aprobación humana. |

## 14. Qué tendría que cumplirse para pasar de piloto a MVP comercial

| Área | Condición necesaria |
|---|---|
| Seguridad | Resolver vulnerabilidades y cerrar riesgos críticos. |
| RGPD / legal | Textos legales revisados, DPA firmado y responsabilidades claras. |
| Retención | Política formal de “solo metadatos” y borrado operativo implementados. |
| Privacidad | Persistencia desactivada por defecto o gobernada con cifrado, acceso y borrado. |
| XML | Validador XSD estándar integrado. |
| SES | Evidencia de aceptación en preproducción antes de claims de compatibilidad. |
| QA | Pruebas E2E y pruebas con datos sintéticos representativos. |
| Producto | Mapeador visual o proceso claro para adaptar Excel de distintos clientes. |
| Comercial | Mensaje sin promesas de cumplimiento, sanciones, integración oficial ni automatización no validada. |
| Soporte | Protocolo de onboarding, incidencias y revisión humana. |

### Decisión prudente

Pasar de piloto a MVP comercial solo debería ocurrir cuando haya evidencia técnica, legal y operativa suficiente. Antes de eso, la oferta debe seguir siendo **piloto controlado con datos sintéticos o anonimizados**.
