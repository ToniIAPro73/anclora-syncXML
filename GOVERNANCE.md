# Governance

## Current governance model

Anclora SyncXML is currently maintainer-led.

At this stage, the project is still in pre-MVP / controlled validation, so
technical direction, release readiness and scope decisions are curated
conservatively.

## Decision priorities

When tradeoffs appear, the project should favor:

1. Privacy and data minimization.
2. Security and safe defaults.
3. Honest public documentation over aspirational claims.
4. Reproducibility, auditability and maintainability.
5. Contributor experience where it does not weaken the points above.

## How changes are evaluated

Maintainers will typically look for:

- Clear problem statement.
- Fit with the current roadmap.
- No expansion of legal/compliance claims beyond what is implemented and
  reviewed.
- No use of real guest data in code, tests, docs or issue discussion.
- Appropriate tests or documentation updates when relevant.

## Scope discipline

The project may reject or defer contributions that:

- Introduce compliance or official-integration claims not backed by code and
  audit evidence.
- Increase handling of sensitive data without corresponding safeguards.
- Expand product surface substantially before the current privacy-first core is
  stable.

## Maintainer responsibilities

Maintainers are expected to:

- Keep public documentation accurate.
- Review privacy and security implications of proposed changes.
- Use synthetic or anonymized examples in public collaboration.
- Keep issue and PR triage respectful and actionable.

## Community expectations

Contributors are expected to follow:

- [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md)
- [CONTRIBUTING.md](CONTRIBUTING.md)
- [SECURITY.md](SECURITY.md)

## Governance evolution

If the project gains sustained external contribution, this document can evolve
to describe broader review rights, release roles, ownership areas or lightweight
RFC processes.
